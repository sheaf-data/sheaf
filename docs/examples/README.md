# Example Sheaf reports & recipes

This directory holds two kinds of example: **walkthrough directories** (each with
its own README explaining a recipe end-to-end) and **standalone recipe configs**
(a `*-coverage-config.textproto` + `*-coverage-rules.textproto` pair you copy into
a target checkout). Many recipes render into [`example-reports/`](../../example-reports/).

## Walkthrough directories

Each of these has its own README:

| Directory | What it shows |
|---|---|
| [`envoy-coverage/`](envoy-coverage/) | End-to-end recipe scanning [envoyproxy/envoy] with the `protocpp` adapter, scoped to the eight v3 xDS management-plane services. `validate.sh` runs the full pipeline and diffs element counts against a proto-descriptor ground truth. |
| [`self-scan/`](self-scan/) | Sheaf dogfooding itself: the `sheaf.textproto` + source map Sheaf uses to scan its *own* cobra CLI surface against its own tests (`gotest`), docs, and worked examples. The canonical "complete config for a cobra-style CLI". |
| [`sheaf-bot-demo/`](sheaf-bot-demo/) | The PR-review bot (`sheaf review` CLI + `review_pr` MCP op) over a synthetic base/head "widgets" project, showing both surfaces produce the same review comment. |
| [`sheaf-validate-scan/`](sheaf-validate-scan/) | A Claude Code skill that deep-validates any scan — classifying each attribution as TP / FP / ambiguous and searching for false negatives. (Superseded for onboarding by `sheaf verify` + the `sheaf-onboard` skill; kept for standalone audits.) |
| [`sheaf-onboard/`](sheaf-onboard/) | A platform-agnostic skill that onboards a repo to Sheaf and produces a first report a team can trust — automating the config, then adversarially verifying every top-line number against disk before the report is shown. |
| [`sheaf-build-adapter/`](sheaf-build-adapter/) | A platform-agnostic skill that authors one new adapter to close one evidenced coverage gap, proving it with `sheaf verify`. The code-writing counterpart `sheaf-onboard` hands off to. |

## `fuchsia-io-coverage/`

A live coverage report generated by Sheaf against the `fuchsia.io` FIDL library plus the in-tree VFS test corpus. Produced with:

```sh
sheaf report --repo <fuchsia> --format html --output docs/examples/fuchsia-io-coverage
```

**Open the report:** [`fuchsia-io-coverage/index.html`](fuchsia-io-coverage/index.html)

### What's inside

| Page | Purpose |
|---|---|
| `index.html` | Sortable element table + 5 summary cards (Elements, Documented, Tested, Findings, Libraries). 212-row filterable table with color-coded test/doc/example counts and missing-category pills. |
| `findings.html` | All findings sortable + filterable. 248 entries: 201 MISSING_IN_CATEGORY, 43 DOCUMENTED_UNTESTED, 3 TESTED_UNDOCUMENTED, 1 THIN_REFERENCE. |
| `elements/*.html` | One page per ContractElement (212 total). Rich detail: kind/library/location, doc excerpt, relationship list, per-bucket documentation refs with substance pills + canonical fuchsia.dev URLs, test refs, missing-categories list, element-scoped findings. |

### Top elements by test attribution

| Element | Tests | Docs | Notes |
|---|---|---|---|
| `fuchsia.io/Node` | 74 | 1 | Protocol surface; tests via implements-map (NodeConnection) |
| `fuchsia.io/Directory` | 69 | 1 | Composes Openable + Node + AdvisoryLocking |
| `fuchsia.io/File` | 69 | 1 | Composes Readable + Writable + Node + ... |
| `fuchsia.io/Operations` | 19 | 1 | Bits type for capability flags |
| `fuchsia.io/File.Read` | 3 | 0 | Tested via name-token match; doc inherited via composition |
| `fuchsia.io/Directory.Open` | 1 | 2 | SUBSTANTIVE doc (176 + 41 words); 1 unit test in service_tests.cc |

### How it's generated

The same scan that produces this report also serves data via `sheaf serve` (the MCP server) and `sheaf gaps` (the findings CLI). The HTML is one of several views over the same in-memory CoverageProfile index.

Config used: see [`fuchsia-io-coverage-config.textproto`](fuchsia-io-coverage-config.textproto).

### Regenerating

```sh
# Adjust the paths to your local Fuchsia checkout.
cp docs/examples/fuchsia-io-coverage-config.textproto /path/to/fuchsia/sheaf.textproto
cp docs/examples/fuchsia-io-coverage-rules.textproto /path/to/fuchsia/categorization-rules.textproto
sheaf report --repo /path/to/fuchsia --format html --output docs/examples/fuchsia-io-coverage
```

End-to-end time: ~5 seconds (ingest 5s, index 40ms, analyze ~0, render <1s).

## `ffx-coverage-config.textproto` + `ffx-coverage-rules.textproto`

Companion config for scanning Fuchsia's `ffx` CLI. The contract surface is the
`ffx` subcommand/flag tree; tests are attributed through the `rust-test` adapter,
including ffx subprocess invocations extracted from the test corpus.

### Run

```sh
cp docs/examples/ffx-coverage-config.textproto /path/to/fuchsia/sheaf.textproto
cp docs/examples/ffx-coverage-rules.textproto /path/to/fuchsia/categorization-rules.textproto
sheaf report --repo /path/to/fuchsia --format html --output <out-dir>
```

## Other recipe configs

The same `*-coverage-config.textproto` + `*-coverage-rules.textproto` shape covers
the other surfaces shipped alongside this directory — the broader Fuchsia families
(`fuchsia-driver-framework-*`, `fuchsia-ui-composition-*`, `fuchsia-ui-gfx-*`), the
Pigweed modules (`pigweed-pw_log`, `pigweed-pw_rpc`, `pigweed-pw_transfer`), and the
GitHub CLI (`gh-*`). Copy the pair into a checkout and run `sheaf report` (or
`sheaf scan` + `scanner`) as in the recipes above.
